# Reference Pack Catalogue — Plan

A curated, versioned, regularly-refreshed library of best-in-class
ObservabilityPack manifests for the platforms engineers actually run
(IBM MQ, Kafka, Kong API, IBM Integration Server, MFT, Postgres, Redis,
Kubernetes, …). The studio fetches the catalogue and lets users pick a
reference pack as Pack B; Compare + Traceability then show exactly where
the user's observability falls short of the gold standard for that
specific product.

This document describes:
1. What the catalogue is and where it lives
2. How packs get into it (quality bar + contribution flow)
3. How the studio consumes it
4. The Cowork agent definition that keeps it fresh weekly

---

## 1 · The catalogue

### Repository

New repo: **`MoebiusX/otel-observability-pack-catalogue`**

Separate from the spec (`otel-observability-pack`) and the studio
(`otel-observability-pack-studio`) so its release cycle, contributor
policy, and CI can evolve independently — and so a future organization
that disagrees with our curation can fork it without forking the spec.

### Layout

```

├── CONTRIBUTING.md                        # quality bar + PR template
```

### `CATALOGUE.json` shape

The single file the studio fetches. Indexable, cacheable, diff-friendly.

```json
{
  "schemaVersion": 1,
  "catalogueVersion": "2026.24.1",
  "updated": "2026-06-13T00:00:00Z",
  "categories": [
    {
      "id": "messaging",
      "label": "Messaging & Streaming",
      "packs": [
        {
          "id": "ibm-mq",
          "label": "IBM MQ",
          "vendor": "IBM",
          "productVersion": "9.4.x",
          "packVersion": "1.2.0",
          "tier": "tier-1",
          "path": "packs/messaging/ibm-mq/pack.yaml",
          "evidence": "packs/messaging/ibm-mq/EVIDENCE.md",
          "owners": ["@krystalinex/curators"],
          "lastReviewed": "2026-06-07T00:00:00Z",

          "tags": ["queue-manager", "channels", "dlq"]
        }
      ]
    }
  ]
}
```

## 2 · Quality bar

A pack is catalogue-worthy only when:

| Criterion              | Rule                                                                       |
|------------------------|----------------------------------------------------------------------------|
| Schema valid           | Passes `validate-canonical` against vendored v1.2 schema                   |
| Conformance ≥ tier-2   | MUST = 100% for declared tier-2 (or 100% for whatever tier it claims)     |
| Evidence required      | Every non-obvious choice cites a vendor doc, OTel semconv, or RFC         |
| Reproducible           | A reader could rebuild the pack from EVIDENCE.md + vendor docs            |
| Vendor-version pinned  | `metadata.bindings.target` names the product + version range covered      |
| Licensed for reuse     | EVIDENCE.md sources must be public; no internal-only references           |

CI gate at PR time enforces the mechanical bits (schema + conformance);
human review enforces the editorial bits (evidence + reproducibility).

## 3 · Studio integration

### New server endpoints

```
GET  /api/catalogue                    # proxies CATALOGUE.json with 1h cache
GET  /api/catalogue/:packId            # proxies packs/<path>/pack.yaml
GET  /api/catalogue/:packId/evidence   # proxies EVIDENCE.md (rendered)
```

The proxy sits behind a `CATALOGUE_URL` env var (default
`https://raw.githubusercontent.com/MoebiusX/otel-observability-pack-catalogue/main`)
so a regulated organization can point at their own fork without touching code.

### UI placement

Two surfaces:

1. **In the Pack B picker** — alongside the existing "uploaded" and
   "example" packs, add a "Reference (curated)" section. Picking one
   pulls the YAML over HTTPS and adopts it through the existing
   uploaded-pack code path (so it gets a deterministic content-id, lands
   in state.catalog, and plugs into Compare / Traceability without any
   new state plumbing).

2. **On the home screen** — after MCP connect succeeds and the user
   has a draft pack, surface a "Benchmark against the reference for
   <inferred product>" CTA. If the MCP attests to Kafka brokers,
   suggest `packs/messaging/kafka`. Inferred from `system_topology` or
   from `metadata.bindings.target` in the user's draft.

### Compare/Traceability framing for reference packs

When Pack B is a catalogue pack, the bucket semantics shift slightly:

| Bucket                 | Meaning when B is a catalogue reference                            |
|------------------------|--------------------------------------------------------------------|
| Aligned                | You're doing what best-in-class does                               |
| Declared, not verified | You wrote the right intent; the reference confirms what to deploy |
| Verified, not declared | The reference says you don't need this; consider removing          |
| Stale declaration      | Your intent diverges from best-in-class — review                   |

A reference pack on the right is a *target*. The user moves their pack
toward it through deploys + manifest edits.

## 4 · Cowork agent — the weekly refresh

A scheduled Cowork agent that keeps the catalogue current.

### Schedule
Weekly (`0 6 * * 1` — Monday 06:00 UTC). Cron in
`.github/workflows/weekly-refresh.yml` triggers a `workflow_dispatch`
on the catalogue repo, which calls the Cowork agent via webhook.

### Agent loop

```
for each pack in CATALOGUE.json:
  1. Fetch the upstream product's docs (vendor URL list in version.json)
  2. Fetch the latest OTel semantic conventions for the product class
  3. Compare with the pack's current declared shape
  4. If meaningful gap (new tools, new conventions, deprecated practice):
       - Open a PR on the catalogue repo:
         · diff in pack.yaml
         · addendum to EVIDENCE.md citing the new source
         · CHANGELOG.md entry
         · request review from owners listed in CATALOGUE.json
  5. Else update version.json.lastReviewed and continue

then:
  6. Check the BACKLOG.md for requested products with no pack yet
  7. Pick the highest-voted request
  8. Research the product (vendor docs, community packs, OTel collectors)
  9. Generate a candidate pack following the canonical spec
  10. Open a PR adding it to packs/<category>/<id>/ with full evidence
```

### Agent prompt template (Cowork skill definition)

```yaml
name: catalogue-refresh
description: |
  Keep the ObservabilityPack reference catalogue fresh. Each run audits
  every cataloged pack against upstream vendor docs + OTel semantic
  conventions, opens PRs for meaningful gaps, and proposes new packs from
  the request backlog.

schedule: "0 6 * * 1"           # Monday 06:00 UTC

permissions:
  github:
    - read: MoebiusX/otel-observability-pack-catalogue
    - write: MoebiusX/otel-observability-pack-catalogue (PRs only, no direct main pushes)
  network:
    - vendor doc URLs declared per pack in version.json
    - opentelemetry.io
    - github.com / pkg.go.dev / docs.confluent.io / docs.ibm.com (read-only)

tools:
  - WebFetch                    # vendor docs
  - WebSearch                   # discovery for new pack research
  - Bash (gh CLI)               # PR creation + review-request
  - validate-canonical          # invoke the studio's validator on candidate packs

inputs:
  - CATALOGUE_REPO (default: MoebiusX/otel-observability-pack-catalogue)
  - BACKLOG_PATH (default: BACKLOG.md)
  - MAX_PRS_PER_RUN (default: 5)
  - DRY_RUN (default: false — set true for the first deployment)

output:
  - GitHub PRs on the catalogue repo, each titled
    "refresh(<pack-id>): <one-line summary>"
  - one summary issue per run with the list of PRs opened + reviewers
    pinged + the products audited without changes
```

### Failure modes & guards

- **Hallucinated evidence.** Every claim in EVIDENCE.md must link to a
  source URL the agent actually fetched. Agent is required to attach
  the fetched URL + timestamp + content excerpt to the PR description.
- **Schema drift.** Agent runs `validate-canonical` locally before
  opening the PR. PR check re-runs it; mismatch fails the PR.
- **Vendor doc rate limiting.** Agent caches fetches for 7 days
  (matches the refresh cadence).
- **Runaway PR generation.** `MAX_PRS_PER_RUN` cap. Backlog grows but
  doesn't flood reviewers.
- **Quiet weeks.** A run that opens 0 PRs still posts a summary issue
  with "audited N packs, no meaningful changes" so the system feels
  alive, not dead.

## 5 · MVP — what ships first

To get the loop going without front-loading curation work:

### Sprint 1 (week of 2026-06-15, post-FOST)

- Seed with **3 packs** the team can hand-author confidently:

  Server) — either hand-authored OR agent-drafted

## 6 · Open questions

| Question | Why it matters | Default we'd ship |
|---|---|---|
| Should reference packs version with the product (Kafka 3.6 vs 3.7) or be evergreen? | First means more curation work; second means less precision | Evergreen with `productVersion: "3.x"` ranges; add per-version packs only when behavior diverges materially |
| Single org owns curation, or federated? | Single = consistent quality; federated = scales | Single owner (MoebiusX) for v1.0; federated namespaces (`packs/org/<org>/<product>/`) in v2.0 |
| Paywall / sponsor model for the catalogue? | Funds sustained curation | Open + MIT for now; revisit if Cowork agent costs balloon |
| Telemetry on what users benchmark against most? | Roadmap signal | Opt-in only; never tied to identity. A POST to a /telemetry endpoint with `{packId, hash, timestamp}` |
| Catalogue mirror in spec repo for atomicity? | If spec changes, references break | Keep catalogue separate; pin to a specific spec version in `version.json` and refresh manually on breaking changes |

## What this unlocks

- **First-time visitor flow**: open studio → connect MCP → "your platform runs Kafka and IBM MQ — here's how IBM MQ at tier-1 is supposed to look" → 30 seconds to a gap report → adopt.
- **Long-term assurance**: the user's pack stays benchmarked against the latest best practice automatically. When OTel ships new semconv for Kafka, the Cowork agent surfaces it within a week.
- **Network effects**: every well-curated pack in the catalogue increases the studio's value proposition. The catalogue is the moat.
